1.1 组件的三个生命周期

1.2.1 装载阶段（执行顺序由上到下）

getDefaultProps getInitalState

getDefaultProps 以及 getInitState 是 React 早期使用 React.createClass 定义组件时，初始化化 props，state 时使用。目前 React 官方推荐使用 class extends 继承方法实例化组件。所以这两个函数先不予了解更多。

constructor

使用 class extends 实例化组件中使用该函数进行初始化过程。

其中 super() 的意思大概为继承了一个类的引用。再通俗点来说就是 super(props) 之后，整个函数内都可以使用 this.props 了。

紧接着就是初始化state 。对 this.state 赋值一个对象就完成了 state 初始化

componentWillMount

componentWillMount 该函数可以再在浏览器端或者服务器端使用，但是浏览器端使用基本不多，因为 compomentWillMount 在 render 之前，此时没有进行任何渲染，所以这里能干的事，在 constructor 内都可以进行。可以把这个函数理解成和 componentDidMount 相对应的一个函数，compomentWillMount 和 componentDidMount 两个函数像门神一样一前一后夹着 render 。

需要注意的是 componentWillMount 能够在服务器端调用，要做服务器端渲染(SSR) 时可以在服务器端调用该函数。

render

render 作为最重要的生命周期函数，他应该保持整个函数内部的整洁，所以不要出现 this.setState 等改变外部数据的事情。render return 的是 jsx 标签，之后调用 React 内部的 React.createElement 从而实现 DOM 的渲染，而不是大家想的 render 直接渲染 dom。

componentDidMount

componentDidMount 在 render 之后，dom结构渲染完毕，组件以及挂载到网页上之后执行的函数。需要注意的第一点是，并不是单个组件 render 结束后立刻调用该函数。而是在网页第一次渲染结束后，统一调用该函数。在使用一些其他第三方库时，可以在此处进行操作，例如: jQuery，因为此时 dom 树已经搭好，可以随意获取 dom 元素。另外发送请求一般也是在该函数中进行，具体会在文章后文中讨论这个问题。

1.2.2 更新阶段

componentWillReceiveProps (外部 props 发生改变时)

当父级 render 时，子组件就会跟接着触发该函数。该函数参数为即将要更新的 props ，所以可以在此函数内 this.setState ，紧接着进入 shouldComponentUpdate。 需要注意的一点为，在此函数内改变自身 state 并不会陷入死循环（更新阶段该函数只调用一次。

⚠️ 更新阶段除此函数内可以改变 state，不会造成死循环。其余都会造成死循环。

shouldComponentUpdate （自身 state 发生改变）

shouldComponentUpdate 函数有两个参数，第一个为新的 state， 第二个为旧的 state。它是除了 render 外，唯一需要返回值的。 默认范围 true 。如果返回值为 ture 则继续，否则则不更新。所以该函数定义了当发生更新时应该渲染什么，不渲染什么。利用好本函数，可以很高的提高整个 React 项目的性能。

componentWillUpdate

当 shouldComponentUpdate 返回值为 true 时，会进入调用该函数。

render

同挂载阶段。

componentDidUpdate

同 componentDidMount ，可以在更新结束，重新渲染 dom 后再对 dom 进行操作。

1.2.3 卸载阶段

componentWIllUnmount

当组件卸载时会调用，可以进行清除定时器，缓存等。

小结：

  • 当一个组件正常使用时，挂载和卸载阶段的生命周期函数一般只执行一次，重复执行的都在 更新阶段。但是 更新阶段 只有 componentWillReceiveProps 中能使用 this.setState 。
  • 在挂载阶段和更新阶段，有一个共同的函数，那就是 render。

什么时候让组件重新刷新？

组件重新加载有两种方法：1. 改变自身 state，或者 render 中直接使用外部 props(因为render为两个阶段共有函数) 。但是要注意的是各个生命周期的执行顺序以及次数。

举个栗子：

  • 该组件只挂载阶段的初始化state使用了props，之后外部props改变，内部的state并不会改变。因为挂载阶段只会执行一次。

    所以该组件虽然外部 props 发生改变，但是内部除了 constructor 以外并没有使用到 props，所以没有进行更新。

    本质上来说其实 props 发生了改变，组件会走更新的生命周期，但是这个 demo 中并没有在更新的声明周期中有任何操作。

再举个栗子：

  • 这次会更新是因为什么呢？因为render在挂载和更新阶段有存在，当外部props发生改变时，走更新的声明周期，再次执行到 render 时候就更新了 dom。当我们在初始化中需要使用props 时，当 props 发生改变时， 可以在 componentWillReceiveProps 中使用 this.setState ，是比较好的选择。

为什么一般都在 componentDidMount 中发送请求？

首先我们要清楚这个顺序： componentWillMount > render > `componentDidMount 。

上面的生命周期中说过，componentWillMount 可以在服务器端使用，所以第一次渲染可以由服务器调用 componentWillMount 完成渲染。之后再执行 render 先将 dom 结构渲染完成，再在 componentWillMount 中完成补充内容的操作。在没有服务器端渲染的情况下，如果请求是异步的，在 componentWillMount 还是 componentDidMount 中发送请求是没有差别的，因为都不会阻挡你组件的挂载流程。如果是同步的，在 componentWillMount 中请求的话会造成挂载过程的阻塞，因为第一次render推迟执行，从而导致白屏卡顿，或者可能多次进行调用，发送多个请求。如果在 componentWillMount 中进行同步的请求的话，可以在第一次 render 中渲染一些 loading 画面等等。之后等数据返回之后再展现内容，可以提高用户体验。
